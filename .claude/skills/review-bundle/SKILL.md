# review-bundle: core/view 分離レビューバンドルの生成

正本は METHODOLOGY.md（本スキルと同梱 or hub 層）。本書は実行手順への展開。

## 手順

1. **対象確定**: レビュー待ち項目を棚卸しし、1レビューイベント = 1バンドルに統合する（乱立させない）
2. **core.md を書く**（必ず view より先）:
   - 場所: `docs/review-bundles/<date>_<topic>/core.md`
   - メタ: title / date / status: レビュー待ち / 対象一覧 / **as_of: 現在の HEAD ハッシュ** / changelog
   - セクション: findings / open_points / confirmations / risks
   - 全項目に 🟦(依頼者・文書・実測由来) / 🟨(自分の推測) と 確定/仮説/未確認、source_refs(path＋section)
   - open_points: 問い / 背景(なぜこの判断が発生するか) / 推奨 / 却下時(代替の実在を検証して書く) / depends_on(任意)
   - **RP は 10 個以内**。超過分は confirmations へ降格
3. **views/review.md を派生**:
   - 30秒サマリー →「判断が必要な点は N 個」表（# | 問い | 背景 | 推奨 | 却下時）
   - depends_on があればレビュー順を明示
   - 確認のみ / リスク要点 / レビュー後の流れ
   - 判定返却テンプレ（**先頭に「全 RP 一括 ✅」の1行**、続けて RP 別の ✅/✏️/❌＋コメント欄）
4. **付随物のコミット**: 生成時の所感・改善提案・計測値（正味時間・RP数）もバンドル配下に置く
5. commit & push し、バンドルパス・RP 個数・想定レビュー時間を報告して終了

## 厳守事項

- **原本を改変しない**（参照のみ）。判定は実行しない（推奨提示まで。ゲート判定は不可侵）
- 根拠を確認できなかった記述は**未確認**とし、確信があるように書かない
- 修正依頼（✏️/❌）を受けたら **core を直してから view を再派生**。同一ディレクトリで改訂し changelog に追記
- 既存のレビューキュー（checklist 等）があれば、そこから本バンドルへの単方向リンクを1行追加するのみ。ad-hoc なレビューガイドは作らない
- 軽量原則: .md のみ。HTML 化・スクリプト化・凝った整形はしない
